# Policies

This document is written for both humans and coding agents. Treat every policy below as a default rule for all new code and refactors unless an existing project constraint makes it impossible. If two policies appear to conflict, choose the option that keeps the codebase more type-safe, more consistent with existing patterns, and easier to maintain.

## 1. Import GraphQL helpers from `__generated__`

- Always import `gql` from `__generated__`.
- Preferred import:

```ts
import { gql } from '__generated__';
```

- Do not import `gql` from `@apollo/client`, `graphql-tag`, or any other package.
- Reason: the generated `gql` helper is connected to this project's GraphQL code generation setup and preserves typed documents.

## 2. Rely on generated GraphQL types

- Prefer types generated by GraphQL codegen over handwritten TypeScript interfaces for query results, mutation results, variables, fragments, and nested GraphQL objects.
- When data comes from a GraphQL operation, first look for an existing generated type before creating a custom type.
- Avoid duplicating schema shapes manually. Duplicated types drift over time and weaken type safety.
- If you need a derived UI shape, derive it from the generated type instead of redefining the original server shape from scratch.

## 3. Do not create components inside the `/routes` folder

- Files under `src/routes` should focus on routing concerns only:
  - route definitions
  - route-level data loading or GraphQL operations when appropriate
  - route composition
  - wiring page-level components together
- Reusable UI and feature logic should live outside `src/routes`, usually in `src/components`, `src/features`, or another shared module.
- If a JSX block inside a route becomes large, reusable, or stateful, extract it into a component outside the routes folder.
- Small route-local helpers are acceptable only when they are truly route-specific and not meaningful as standalone components.

## 4. Keep each `.tsx` file under 1000 lines

- No `.tsx` file should exceed 1000 lines of code.
- This is a hard maintainability limit, not just a style preference.
- If a file grows too large, split it by responsibility:
  - presentational components
  - hooks
  - GraphQL queries or mutations
  - utility functions
  - constants
- Prefer smaller, focused modules before the file approaches the limit. Do not wait until a file becomes unmanageable.

## 5. Do not leave lint or TypeScript errors

- New changes must not introduce ESLint errors, TypeScript errors, or obvious type regressions.
- Code should be valid under the project's existing `lint` and `typecheck` expectations.
- Do not silence errors unless there is a real, documented reason and no safer alternative.
- Avoid using comments like `@ts-ignore` as a shortcut. If suppression is truly unavoidable, use the narrowest possible suppression and explain why.
- Test if you have errors by running `bun run lint` (to check for eslint errors) and `bun run typecheck` (to check for typescript errors).

## 6. Do not create new Apollo clients

- Use the existing Apollo client setup already established by the project.
- Do not instantiate additional Apollo clients in features, components, hooks, or routes unless there is an explicit architectural decision already present in the codebase.
- If a new data access pattern is needed, integrate it with the current client instead of creating a parallel client configuration.
- Reason: multiple Apollo clients usually create inconsistent caching, duplicated configuration, and harder-to-debug behavior.

## 7. Do not use `any`

- `any` is prohibited in application code.
- Always prefer a real type, a generic, a union, a discriminated union, or a generated GraphQL type.
- If a value is not fully known yet, model the uncertainty explicitly instead of erasing it with `any`.
- If you are tempted to use `any`, treat that as a signal to improve the surrounding type design.

## 8. Avoid `unknown as` unless it is truly necessary

- Do not use `unknown as SomeType` as a routine shortcut to force values through the type system.
- Double casting should be a last resort, only when:
  - the runtime shape is already guaranteed elsewhere, and
  - there is no cleaner typing strategy available
- Before using `unknown as`, prefer:
  - narrowing with runtime checks
  - better generic constraints
  - Zod validation
  - generated types
  - helper functions with explicit return types
- If `unknown as` is unavoidable, keep it isolated and document why the cast is safe.

## 9. Use `logger` instead of `console`

- Do not add `console.log`, `console.error`, `console.warn`, or similar direct console calls.
- Use the project's shared `logger` utility instead.
- Prefer structured, intentional log messages that help with debugging without creating noise.
- Use the appropriate log level such as `logger.log`, `logger.debug`, `logger.warn`, or `logger.error`.
- If editing old code that still uses `console`, migrate it to `logger` when it is reasonable to do so within the current change.

## 10. Use Zod for type validation

- When validating external, dynamic, or user-controlled data, use Zod schemas.
- Common cases include:
  - form data
  - URL params or search params
  - local storage data
  - API responses that are not already fully typed and trusted
  - values crossing untyped boundaries
- Do not rely on TypeScript types alone for runtime validation, because TypeScript only checks at compile time.
- Prefer defining a Zod schema once and inferring the TypeScript type from it when both runtime validation and static typing are needed.

## 11. Use Zustand for global state management

- For shared client-side global state, use Zustand.
- Do not introduce a different global state library unless the repository already requires it for a specific subsystem.
- Keep Zustand stores focused and domain-based rather than creating one oversized global store.
- Prefer local component state when the state does not need to be shared broadly.
- Use Zustand only for state that is genuinely cross-component, persistent, or difficult to manage through props alone.

## Working Rule for LLMs

- Before adding a new pattern, first check whether the repository already has an accepted way to solve the same problem.
- Prefer consistency with existing project structure over personal preference.
- When in doubt, choose the option that reduces custom typing, avoids duplication, and keeps code easier to review.
